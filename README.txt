📁 תיעוד תהליך עבודה: עדכון קוד, העלאה ל־GitHub, ופריסה ב־Render
1. ביצוע שינוי בקוד (לוקאלית במחשב שלך)
• ערוך את הקובץ (למשל: botly_api.py) בעורך טקסט כמו VS Code / Notepad++ / PyCharm וכו'.
• שמור את הקובץ לאחר השינוי.
2. עדכון השינוי ל־GitHub
פתח את ה־Command Prompt או ה־Terminal בתיקיית הפרויקט שלך (בדיוק כמו שעשית עד עכשיו):
cd "C:\Users\denne\Documents\Combined Botli"
בצע את הפקודות הבאות (בכל שינוי):
1. הוספת הקבצים שהשתנו:
git add .
2. ביצוע Commit עם הודעה (רצוי ברורה):
git commit -m "תיאור השינוי שביצעתי"
3. משיכת עדכונים מהמאגר הרחוק (רשות, אך מומלץ):
git pull origin main --rebase
o פותר בעיות אם מישהו אחר שינה קבצים ב־GitHub.
4. דחיפת השינויים ל־GitHub:
git push origin main
3. הפרויקט ב־Render מתעדכן אוטומטית
•  Render מזהה אוטומטית כל Push ל־GitHub (לרוב תוך דקה עד 2).
• ניתן להיכנס ל־Render Dashboard → לבחור את השירות → ולראות את תהליך ה־Deploy.
• אם יש שגיאה (למשל בקוד), Render יציג אותה בלוגים של השירות.
4. בדיקה שהשירות עלה בהצלחה
• בסיום הפריסה (Deploy), בדוק שהשירות Deployed ושאפשר לגשת אליו ב־URL שלך.
• בדוק את ה־API / האתר — האם הקוד המעודכן רץ? האם הבאג תוקן?

📝 טיפים חשובים:
• כל שינוי קטן דורש את הרצף: add → commit → push.
• שגיאה ב־push ("rejected" או דומה)? קודם כל להריץ git pull origin main --rebase ואז שוב git push.
• לא לשכוח לכתוב הערות ברורות ב־commit — זה יקל עליך לזהות שינויים בדיעבד.
• אם אתה מוסיף קבצים חדשים, ודא שהם כלולים ב־add.
• ל־Render לוקח כמה דקות לעדכן – המתן בסבלנות.

דוגמה לתיעוד שינוי:
רציתי לעדכן את פונקציית ניקוי הקלט:
1. ערכתי את הקובץ botly_api.py.
2. שמרתי את הקובץ.
3. פתחתי טרמינל בתיקיית הפרויקט.
4. הרצתי:
git add .
git commit -m "תיקון פונקציית ניקוי קלט — תומך ביותר סוגי פורמטים"
git pull origin main --rebase
git push origin main
5. נכנסתי ל־Render ובדקתי שהשירות נפרס מחדש.
6. בדקתי שהתיקון עובד.

💡 לסיכום
כל תהליך שינוי ושדרוג עובר ב־4 שלבים עיקריים:
עריכה → Commit + Push ל־GitHub → פריסה אוטומטית ב־Render → בדיקה
כך אתה תמיד שולט בקוד, מגובה אותו, וכל שינוי חוזר על תהליך זה — אין לקפוץ שלבים.
